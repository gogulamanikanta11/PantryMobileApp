# Smart Pantry Mobile App - Backend Specification & Security Design

This specification outlines the data schemas, access controls, input validation summaries, and endpoints list representing the API layer of the Smart Pantry Mobile App.

---

## 1. Endpoint Inventory

The backend database operations and API calls are segmented into five groups: **Authentication**, **User Profiles**, **Pantry Data**, **Shopping List Data**, and **AI Services**.

| Method | Endpoint / SDK Operation | Group | Auth Required | Description |
| :--- | :--- | :--- | :--- | :--- |
| `POST` | `createUserWithEmailAndPassword` | Authentication | No | Register new email/password credentials |
| `POST` | `signInWithEmailAndPassword` | Authentication | No | Authenticate user using email and password |
| `POST` | `signInWithPhoneNumber` | Authentication | No | Request SMS verification code (OTP) |
| `POST` | `confirmationResult.confirm` | Authentication | No | Confirm SMS verification code to sign in |
| `POST` | `sendPasswordResetEmail` | Authentication | No | Send email to reset user password |
| `POST` | `signOut` | Authentication | Yes | Invalidate local session state |
| `GET` | Firestore `/users/{uid}` | User Profiles | Yes | Read registered profile information |
| `SET` | Firestore `/users/{uid}` (merge: true) | User Profiles | Yes | Create or update user profile parameters |
| `GET` | Firestore `/pantry` | Pantry Data | Yes | Query all user's pantry items |
| `POST` | Firestore `/pantry` | Pantry Data | Yes | Append a new pantry record |
| `DELETE`| Firestore `/pantry/{id}` | Pantry Data | Yes | Remove a pantry record by document ID |
| `GET` | Firestore `/shoppingList` | Shopping List | Yes | Query user's current shopping list |
| `POST` | Firestore `/shoppingList` | Shopping List | Yes | Append a new shopping list record |
| `DELETE`| Firestore `/shoppingList/{id}` | Shopping List | Yes | Remove shopping list record by document ID |
| `POST` | `https://openrouter.ai/api/v1/...` | AI Services | Yes | Generate recipes via OpenRouter proxy |
| `POST` | SDK `gemini-2.0-flash` | AI Services | Yes | Generate recipes via Gemini direct API |
| `GET` | Local mock evaluation | AI Services | No | Static fallback recipe builder |

## 3. Role Matrix Documentation

The application implements client-side views mapped to Firestore target security constraints. Since authentication is globally managed, access is structured using the following matrix:

| Endpoint Group | Unauthenticated Visitor | Authenticated User | Admin / Owner |
| :--- | :---: | :---: | :---: |
| **Authentication (Sign Up/Login)** | ✓ Read/Write | ✗ Restricted | ✗ Restricted |
| **User Profile Document (`/users/{uid}`)** | ✗ Denied | ✓ Own Profile Only | ✓ Full Control |
| **Pantry Inventory Collection (`/pantry`)** | ✗ Denied | ✓ Read/Write Own | ✓ Full Control |
| **Shopping List Collection (`/shoppingList`)**| ✗ Denied | ✓ Read/Write Own | ✓ Full Control |
| **AI Recipe Generator (OpenRouter/Gemini)** | ✗ Denied | ✓ Read/Write (via active token) | ✓ Full Control |

### Firestore Access Rules Integration (Recommended Pattern)
To align the security definitions above with the database backend, the following Firestore Security Rules pattern is applied:
```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // User Profile security: Users can only write/read their own matching UID profile document
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
    
    // Pantry security: Active authenticated users can list and update inventory
    match /pantry/{itemId} {
      allow read, write: if request.auth != null;
    }
    
    // Shopping List security: Active authenticated users can query list
    match /shoppingList/{itemId} {
      allow read, write: if request.auth != null;
    }
  }
}
```

## 4. Input Validation Summary

The application performs validation checks before writing documents or calling AI integrations to ensure data sanitization:

### 4.1 Authentication Forms
*   **Email Input (`app/register.tsx`):** Trims trailing spaces. Checked against standard Firebase email formats. Returns `auth/invalid-email` on mismatch.
*   **Password Input (`app/register.tsx`):** Local verification forces input string length to be $\ge 6$ characters before sending network request to avoid unnecessary latency or service errors.

### 4.2 Pantry Entry Form (`app/additem.tsx`)
*   **Item Name:** Strictly required. Throws local `Alert` dialog if empty.
*   **Quantity (Stock):** Mapped to numeric keyboard layout. Parsed to string format on submit.
*   **Expiry Date:** Text input parsing. Fallback to `"Unknown"` if omitted.

### 4.3 Profile Entry Form (`app/profile.tsx`)
*   **Full Name:** Input value trimmed. Required to trigger saving updates.
*   **Phone Number:** Set to `phone-pad` numeric input to limit characters.
*   **Email Address:** Read-only component to prevent discrepancies between login authentication emails and Firestore profile metadata.
